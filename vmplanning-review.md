# VM planning review

Companion to [vmplanning.md](./vmplanning.md). Reviews the design, answers the
nine review questions, and lays out a layered path forward.

## TL;DR

Your prewarm-compute-vs-prewarm-secret-authorization split is the load-bearing
insight. Hold it; everything flows from there.

The single conceptual move I'd push you to make explicit is **separate the
delivery mechanism from the trust root**. Most of your options A–H conflate
these, and once you split them out, the design space collapses cleanly.

Verdict, layered:

1. **Right now**: stay on Option A (Clan vars upload) for `compute-lab` and any
   long-lived service VM on Leviathan. Add Option D (persistent guest age key
   under `/var/lib/tenant-system`) for VMs you treat as pets. This is what
   C3D2, DD-IX, and your homelab today already do for non-VM machines. There
   is no shame in it.
2. **Next primitive (Experiment 2 done well)**: build the seed disk (Option E),
   but carry only a **short-lived bootstrap token** on the seed, not a
   long-lived age key. Make the seed disposable per-boot, generated by the
   host at every VM start, on tmpfs.
3. **When tenants happen**: layer Option F (a small metadata broker on Janus)
   on top. The seed-token gets exchanged for an actual age key over the
   management network. The seed is "an authentication credential for the
   metadata fetch," not "the secret material."
4. **Don't pursue Option B** (QEMU `credentialFiles`) as a long-term answer.
   It's tempting and elegant, but it's a delivery mechanism that locks you
   into one hypervisor and pretends to be a trust root. It is fine as an
   *additional* delivery option for the QEMU path, never as the only one.
5. **Reject Option H**. Generic prewarmed pool with already-hydrated secrets
   is wrong, and your reasoning for rejecting it is correct.

For SOPS recipients: **per-app/env at scale, per-machine for long-lived pets,
never per-ephemeral-instance**.

## The conceptual move: delivery vs trust root

Pretty much every option A–H is some combination of two independent choices.
Pulling them apart:

**Trust root** — what authorizes a guest to obtain secrets:

- `TR1` — a private age key on persistent guest disk (Option D)
- `TR2` — a private age key on host disk, passed at VM boot (B, some E)
- `TR3` — a cloud IAM identity (GCP service account, AWS instance profile)
- `TR4` — a TPM/vTPM with PCR-bound seal
- `TR5` — a short-lived bootstrap token + a signing authority (mTLS, SPIFFE,
  custom PKI) (most F variants)
- `TR6` — network-enforced "this MAC/IP/bridge is trusted" (a weaker `TR5`)

**Delivery mechanism** — how a key/token/cert gets into the guest:

- `D1` — already on persistent disk (`TR1`'s natural delivery)
- `D2` — SSH push from operator (clan vars upload)
- `D3` — fw_cfg / SMBIOS / hypervisor sideband (QEMU credentialFiles, Cloud
  Hypervisor `platformOEMStrings`)
- `D4` — read-only block device / config drive (seed disk, cloud-init NoCloud,
  ConfigDrive)
- `D5` — network metadata service (MMDS, GCE metadata server, custom HTTP
  endpoint)
- `D6` — application-layer fetch from a secret broker (Vault, KMS API, custom)

You can mix any TR with most D's. The mistake is to pick a delivery mechanism
first and let it imply your trust root.

Mapping options A–H to (TR, D):

- **A** = `TR1` + `D2` (persistent + clan SSH push)
- **B** = `TR2` + `D3` (host has key, fw_cfg delivers)
- **C** = `TR2/3` + `D3/D4` (host plaintext)
- **D** = `TR1` + `D1` (persistent on guest disk)
- **E** = `TR2/5` + `D4` (seed disk, contents vary)
- **F** = `TR5/6` + `D5/D6` (broker + token)
- **G** is an operational scoping policy on top of any of the above
- **H** = `TR2/3` + `D?` (snapshot-as-delivery, almost always wrong)

Once you see this, "which option do I pick?" becomes "what trust root am I
willing to maintain, and what delivery is cleanest to ship through?" The two
answers are independent.

## Direct answers to the nine review questions

### 1. Is the prewarm compute/config vs prewarm secret authorization split right?

Yes. This is the single most important framing in the doc — don't lose it.

Every cloud platform that handles this seriously enforces this split, even
when they don't articulate it that way. AWS Lambda SnapStart explicitly
invalidates and refreshes "uniqueness sources" (random seeds, KMS data keys,
ephemeral tokens) on restore precisely because they refused to let the
snapshot become secret material. Firecracker MMDS is the assignment-time
hydration pattern. KubeVirt's cloud-init-from-Secret is the *anti*-pattern
that a lot of people reach for first; it shifts the trust boundary to k8s
RBAC and feels increasingly wrong as soon as you have multi-tenant.

The split is right. The harder question is operationalizing it, which is most
of the rest.

### 2. What do mature platforms actually do for prewarmed secret-bearing VMs?

In rough decreasing-rigor order:

- **AWS Lambda SnapStart**: prewarms runtime + closure + JITted code,
  snapshots after init but **before any KMS data key request**. On restore,
  refreshes random seed, regenerates uniqueness sources, then function code
  re-fetches from KMS. The snapshot is explicitly not secret material.
- **GCP Cloud Run / Lambda regular**: identity is the runtime IAM identity
  (service account), short-lived token vended by the metadata server. Code
  fetches secrets via Secret Manager at startup using that token. No prewarm
  of secret authorization.
- **EC2**: instance profile + IMDSv2 token + Secrets Manager / Parameter
  Store. Same shape.
- **KubeVirt / Kubernetes pods**: ServiceAccount projected token + Secret /
  ConfigMap volume. The token is the trust root, sometimes bound to specific
  audiences. Secrets are mounted at pod start, not baked into images.
- **Firecracker MMDS-based runners (thpham's pattern)**: per-VM short-lived
  token in MMDS, runner registers, runs once, dies. Token never long-lived.
  VM never reused.
- **C3D2, DD-IX**: persistent NixOS guests with state, long-lived age key per
  guest, sops-nix in the guest. Pet model, your Option D.
- **Skyflake**: deliberately leaves the secrets story to the user; the
  platform handles build-and-deploy, not authorization.

The convergent answer: **no mature platform seriously prewarms secret
authorization across an unscoped pool**. They either accept "prewarm only up
to identity claim" (Lambda, KubeVirt, EC2, Firecracker runners), or "warm
pool is per-app/env" (Option G), or "every VM is a pet" (C3D2 / DD-IX / your
homelab today).

There is no fourth option. This is worth burning into the design discussion.

### 3. Is sops-nix appropriate as the in-guest materializer for prewarmed VMs?

Yes — as the **materializer**, not as the **authorization control plane**.

The cleanest pattern:

- Bootstrap puts an age private key (or KMS access) at a known path
- sops-nix runs, decrypts the files declared in the NixOS config, materializes
  them at `/run/secrets/...`
- Service units `LoadCredential` from those paths

What you don't want is for sops-nix to be the thing that *decides* whether a
particular VM is allowed to access a particular secret domain. That's the
broker / IAM / PKI layer's job. sops-nix is downstream.

A clean unit ordering, sketched closer to ready-to-paste than what's in your
doc:

```nix
sops.age.keyFile = "/run/bootstrap/age-key.txt";

systemd.services.fetch-bootstrap-key = {
  description = "Fetch age key from metadata broker";
  unitConfig.DefaultDependencies = false;
  before = [ "sops-install-secrets.service" "sysinit.target" ];
  wantedBy = [ "sops-install-secrets.service" ];
  after = [ "network-online.target" ];
  wants = [ "network-online.target" ];
  serviceConfig = {
    Type = "oneshot";
    RemainAfterExit = true;
    PrivateTmp = true;
    ExecStart = "${pkgs.writeShellScript "fetch-bootstrap-key" ''
      set -euo pipefail
      install -d -m 0700 /run/bootstrap
      token=$(cat /run/seed/token)
      ${pkgs.curl}/bin/curl --silent --fail \
        --cacert /etc/ssl/janus-ca.pem \
        -H "Authorization: Bearer $token" \
        https://janus.lan/v1/age-key \
        > /run/bootstrap/age-key.txt
      chmod 0400 /run/bootstrap/age-key.txt
    ''}";
  };
};

systemd.services.sops-install-secrets = {
  after = [ "fetch-bootstrap-key.service" ];
  requires = [ "fetch-bootstrap-key.service" ];
};
```

This composes cleanly. sops-nix never knows where the key came from. The
broker / seed-disk / whatever is interchangeable.

### 4. Safest, operationally cleanest bootstrap pattern?

Depends on the workload class. There is no single answer. Structure the
homelab around three classes:

**Class A — long-lived service pets** (most of `compute-lab`'s likely peers)

- Persistent identity, persistent age key under `/var/lib/tenant-system/age-key.txt`
- Initial provisioning: clan-style (admin generates key, encrypts to admin,
  uploads)
- After that, the VM is a normal NixOS machine with sops-nix
- Snapshots are fine if you accept the snapshot is secret material
- Option D. Boring. Correct. Most C3D2/DD-IX/your-homelab workloads are this
  class.

**Class B — ephemeral on-demand VMs you spin up for a task and tear down**

- No persistent identity per VM
- Seed disk with one-time token, generated by host at spawn time
- Token exchanged for whatever secret the workload needs
- VM dies, token expires, no rotation
- Maps onto Firecracker-runner / Lambda-style. Options E + F + G.

**Class C — multi-tenant prewarm pools**

- Don't try to make these generic
- Each pool is scoped to a single secret domain (one app, one env, one tenant)
- Per-app/env SOPS recipients; pool VMs all decrypt the same files
- Snapshots are secret material — encrypted at rest, scoped to the pool,
  rotated when secrets rotate
- Option G with the right framing

The mistake is trying to design one bootstrap pattern that handles all three.
They have different threat models and different operational characteristics.
A real platform codifies these three patterns separately.

### 5. Which delivery mechanism would I pick?

For the homelab MVP and the scaling-up trajectory:

- **Now**: persistent guest disk (`D1`). Boring, works.
- **Next primitive**: seed disk (`D4`), but carrying only a short-lived
  bootstrap token, not a long-lived age key. The seed is generated by the
  host at every VM start, lives on tmpfs, dies when the VM stops.
- **Don't choose**: QEMU `credentialFiles` (`D3`) as a long-term answer. Fine
  as an additional delivery for the QEMU path, but if you anchor on it,
  you'll regret the day you want Cloud Hypervisor or Firecracker. The
  microvm.nix `credentialFiles` API itself currently throws on non-QEMU
  hypervisors.

The seed disk has an underrated property: it's the same primitive public
clouds use (config-drive, NoCloud datasource). If you build it well, you can
take a homelab guest config and run it in EC2 or GCE with a different
config-drive datasource and it will just work. That's worth a lot for the
"longer-term self-hosted on-demand" trajectory.

### 6. Per-instance, per-app/env, or per-tenant SOPS recipients?

**Per-app/env at scale.** **Per-tenant when tenant isolation matters more
than env isolation** (mostly customer-hosting cases). **Per-instance only
for long-lived pets**.

Per-instance recipients don't compose with prewarming or autoscaling at all.
Every new VM = re-encrypt every file = redeploy. You can't rotate. The
recipient set is unbounded over time. It's a non-starter for any pool-based
model.

The trade-off with per-app/env is "any VM in the pool can read all the pool's
secrets." That's correct *because the pool is the trust boundary*. If two VMs
in the pool shouldn't share a secret, they shouldn't be in the same pool.
This is the same trade-off as a Kubernetes namespace's secrets — within the
namespace, anything that can mount Secrets can read them. It's fine if your
pool boundary matches your trust boundary.

### 7. What state must be scrubbed/refreshed on snapshot restore?

Concrete list, prioritized:

**Must scrub or regenerate before any service starts**:

- `/run/secrets/` — sops-nix output (re-decrypt with current key)
- `/run/credentials/` — systemd credentials directories (regenerate via
  service restart)
- `/etc/machine-id` (regenerate via `systemd-machine-id-setup`)
- `/var/lib/dbus/machine-id` (re-link to `/etc/machine-id`)
- Kernel random seed (`/var/lib/systemd/random-seed`)
- Host SSH keys, if reuse would be a security or operational problem
- DHCP lease state (let it re-lease)
- `/var/lib/<service>/identity` or similar per-instance identity files
- `/run/bootstrap/` if you put a key there
- `/etc/ssh/ssh_host_*` — likely; depends on whether you persist these

**Must refresh in-process state** (i.e., restart these services after restore):

- TLS session caches
- DB connection pools
- Cached OAuth/IAM tokens
- Anything holding a long-lived in-memory secret
- RNGs (re-seed `/dev/urandom` via random-seed regen)
- Anything that called `getrandom()` early at boot for crypto initialization

**Must refresh externally**:

- DNS records pointing at the old VM's IP
- Reverse proxy / load balancer membership
- Service mesh registration
- Monitoring/alerting agents (re-register)

A pragmatic approach: define a `snapshot-restore.target` that depends on these
regeneration units, and have all secret-consuming services depend on the
target. "Restore from snapshot" becomes "boot, with this extra phase first."

The brutal version: **just restart all services after restore**. Most state
worth scrubbing lives in-process and gets regenerated naturally on service
start. The on-disk scrub list is mostly `machine-id` and the random seed.

### 8. Rotation and revocation?

**Rotation**:

- Per-app/env recipient: `clan vars fix <env>` (or sops equivalent)
  re-encrypts files. Roll the deployments. Old VMs lose ability to read once
  their cached key expires (or on next boot).
- Per-instance recipient (pet model): rotate the per-machine age key,
  re-encrypt only that machine's secrets, redeploy.
- Don't try to rotate "live" — accept a redeploy.

**Revocation**:

- Per-app/env: remove the recipient from creation_rules, run `clan vars fix`,
  redeploy. Existing VMs that haven't restarted still have plaintext in
  `/run`, but can't recover after restart. Decide if you need stronger.
- For real revocation (immediate): use a centralized broker (Option F) that
  can refuse new requests. The on-disk decrypted secrets are the long pole —
  those only go away when the VM restarts.
- Stronger: don't materialize secrets to `/run/secrets` for high-rotation use
  cases. `LoadCredential` straight from a fetched-on-each-start path, and
  signal services to reload. This is what most cloud-native services do.

The thing to internalize: **if your secret is decrypted to `/run/secrets`
and a service is using it, you can't revoke that VM's access without
restarting the VM (or the service)**. sops-nix gives you boot-time
revocation, not runtime revocation. If you need runtime revocation, the
design is different — the secret is fetched per-request, not at boot.

### 9. What assumptions in the doc are too home-lab-specific or off?

- **"MicroVM = full NixOS guest with read-only host /nix/store"** — great
  model for tens of VMs. At thousands-of-VMs scale, you want lighter guests
  (just the closure for the app, no full NixOS), and you want the closure
  shared via dedup, not virtiofs from a host. Single-host virtiofs is a perf
  and reliability bottleneck at scale. Fine for homelab.
- **"warm-on-host-boot is the goal"** — fine for homelab. For real prewarm,
  you want pool-and-claim (a pool of pre-booted VMs claimed by workloads as
  they arrive), not fixed assignment of VMs to hosts at host boot. This needs
  a small scheduler. Janus could grow into one, but probably you'd reach for
  Nomad or KubeVirt rather than build it yourself.
- **"Janus owns DHCP/DNS/firewall for everyone"** — fine at this scale. At
  10k VMs you'd scale out (DHCP relay, multiple DNS servers, distributed
  firewall). Don't worry about it.
- **"MAC/IP-based metadata access is enough"** — true for friendly tenants
  (you and friends). False for hostile or untrusted tenants. The moment a
  customer is in the picture you need real PKI / SPIFFE / mTLS / vTPM
  attestation. The doc acknowledges this; just don't pretend it away when
  the moment arrives.
- **"snapshot for warm pools"** — only true for very specific cases where
  you've solved the scrub-on-restore problem. For most NixOS service VMs
  with a warm Nix store and fast init, just boot. The Lambda SnapStart use
  case (cold-start of JVM workloads on a tight latency SLA) is where
  snapshots earn their keep. NixOS service VMs mostly don't need it.
- **"Generic VM" framing in Options G/H** — drop "generic" from your
  vocabulary entirely except as a thing-not-to-build. Pools are scoped.
  Always. The interesting design space is "what is the scope, and how is it
  enforced."
- **`microvm.deploy.sshSwitch` for hot updates** — fine, but for a serious
  platform, the unit of update is "rebuild the VM closure, atomically swap
  the running VM with a fresh one." For pets, sshSwitch is convenient. For
  pools, you don't update — you replace.

## Recommended path forward (concrete, layered)

### Phase 1 — confirm the boring baseline (Experiment 1)

Goal: prove `compute-lab` works end-to-end with normal Clan vars upload,
before adding any new bootstrap mechanism.

Concrete: pick one secret-consuming service (something tiny — an HTTP echo
server with a `Bearer` token in env, behind `LoadCredential`), add it to
`compute-lab`'s NixOS config, run `clan vars generate compute-lab && clan vars
upload compute-lab`, verify it reads from `/run/secrets/...`.

This validates the chain end-to-end with zero new infrastructure. If this is
broken, none of the further phases matter.

### Phase 2 — build the seed disk primitive (Experiment 2 done well)

Goal: a hypervisor-portable, hot-pluggable way to deliver an early-boot
bootstrap blob to a guest.

Sketch (host-side):

```nix
# modules/microvms/seed-disk.nix
{ config, lib, pkgs, self, ... }:

let
  inherit (lib) mkOption types mapAttrs;
  cfg = config.compute.seed;
  assignedTenantNames = self.compute.assignments.${
    config.clan.core.settings.machine.name
  } or [ ];
  assignedTenants = lib.genAttrs assignedTenantNames (n: self.compute.tenants.${n});
in
{
  options.compute.seed = {
    enable = mkOption { type = types.bool; default = true; };
    stateDir = mkOption { type = types.path; default = "/run/microvm-seeds"; };
  };

  config = lib.mkIf cfg.enable {
    fileSystems."${cfg.stateDir}" = {
      device = "tmpfs";
      fsType = "tmpfs";
      options = [ "mode=0700" "size=64M" ];
    };

    systemd.services = mapAttrs (name: tenant: {
      description = "Build seed disk for microvm ${name}";
      wantedBy = [ "microvm@${name}.service" ];
      before = [ "microvm@${name}.service" ];
      unitConfig.RequiresMountsFor = [ cfg.stateDir ];
      serviceConfig = {
        Type = "oneshot";
        RemainAfterExit = true;
        ExecStart = "${pkgs.writeShellScript "build-seed-${name}" ''
          set -euo pipefail
          seed=${cfg.stateDir}/${name}.img
          tmpdir=$(mktemp -d)
          # Mint a one-time bootstrap token, scoped to this VM
          ${pkgs.openssl}/bin/openssl rand -hex 32 > "$tmpdir/token"
          # Sign with host signing key so the broker can verify
          ${pkgs.openssl}/bin/openssl dgst -sha256 -sign \
            "$CREDENTIALS_DIRECTORY/host-signing-key" \
            -out "$tmpdir/token.sig" "$tmpdir/token"
          echo "${name}"   > "$tmpdir/vm-id"
          echo "${tenant.network or "tenant"}" > "$tmpdir/network"
          ${pkgs.dosfstools}/bin/mkfs.fat -F 12 -C "$seed" 1024
          ${pkgs.mtools}/bin/mcopy -i "$seed" "$tmpdir"/* ::
          rm -rf "$tmpdir"
        ''}";
        LoadCredential =
          [ "host-signing-key:${config.clan.core.vars.generators.compute-host.files.signing-key.path}" ];
      };
    }) assignedTenants;
  };
}
```

Guest side (`guest-base.nix`):

```nix
# /run/seed mount, populated by host before VM boot
fileSystems."/run/seed" = {
  device = "/dev/disk/by-label/SEED";
  fsType = "vfat";
  options = [ "ro" "noatime" ];
  neededForBoot = true;
};

# fetch-bootstrap-key.service per the snippet in §3 above
```

Open questions you flagged that this resolves:

- **"How early can the seed be mounted for sops-nix?"** —
  `neededForBoot = true` mounts in initrd. sops-install-secrets runs after
  `local-fs.target`, so the seed is available. ✓
- **"Does this work with QEMU/CH/Firecracker?"** — yes, all three support
  attaching a virtio-blk device. FAT12 is universal. ✓
- **"Long-lived age key vs short-lived token on the seed?"** — short-lived
  token. Decide once, don't go back. The seed is regenerated per boot, the
  token is single-use. The actual age key lives only in the guest's tmpfs,
  never on a snapshotable surface.
- **"How does rotation work?"** — VM restart re-mints a fresh token; the
  broker reissues the current age key. No coordinated rotation needed.

A note on your `microvm.vms.<name>.flake = self` constraint: this design
sidesteps it. The host doesn't need to reach into the guest's NixOS
config — it produces a *file* that the guest mounts at runtime. The guest
declares once that it has a `/run/seed` mount and a fetch service; the host
fills the seed before each VM start. No host-side config injection required.

### Phase 3 — metadata broker (when the homelab grows tenants)

Goal: Janus (or a small dedicated service) issues age keys in exchange for
valid bootstrap tokens.

Shape:

- A small HTTPS service on Janus, listening on the management network only
- Accepts `Authorization: Bearer <token>` where the token was minted by
  Leviathan's seed-disk service and signed with a key the broker has the
  public half of
- On valid token, looks up which tenant/app/env the VM belongs to (from Clan
  inventory), returns the appropriate age key for that scope
- Token is single-use; the broker tracks consumption to prevent replay
- mTLS to pin the broker — the seed includes the broker's CA cert

This is real-but-small infrastructure. The Nix-native way: a tiny Go or Rust
service deployed via Clan, with its own age key storage backed by sops. ~300
LOC for the MVP.

When you do this, you've reinvented "GCE metadata server but locally" —
genuinely the right shape for self-hosted on-demand VMs. It's also what gives
you the path to friendly-customer hosting later.

### Phase 4 — pool scheduling (only if you outgrow Phase 1–3)

If you genuinely need warm pools (most homelabs don't), don't write your own
scheduler. Use Nomad or k3s + KubeVirt. The seed-disk primitive composes with
both: Nomad job spec attaches the seed disk; KubeVirt VM spec mounts a
config-drive Secret. The seed-disk → broker pattern keeps working.

## Where this generalizes

The same design generalizes to any platform with a managed trust root (cloud
IAM + KMS), with one delta: those platforms hand you the trust root for free.

**Cloud-hosted version**: a managed service account + KMS gives you `TR3` +
`TR4` without any work. Per-env IAM grants on per-env KMS keys substitute for
the Janus broker; the metadata server substitutes for the seed disk. sops-nix
with `gcp_kms` (or `aws_kms` / `azure_kv`) recipients is the materializer.
No long-lived age key on disk anywhere — the IAM grant is the credential.

**Homelab version (this doc)**: no free trust root. You have to build one.
Janus is the natural place, but it's actual work, not a one-line config. The
seed disk + signed bootstrap token + broker are the local re-implementation
of "metadata server + IAM + KMS."

So in the homelab, you're working on the *harder* version of the same
problem. The application-layer mechanics — sops-nix as materializer,
`LoadCredential` into services, per-app/env recipients,
fetch-key-before-sops-install — generalize cleanly. The homelab broker and a
cloud KMS are interchangeable plug-ins behind the same shape.

That's the design property to optimize for. Make sure the seed-disk + fetch
hook is clean enough that swapping "talk to Janus" for "talk to a cloud
metadata + KMS API" is a single Nix module.

## Snapshot test (sanity check)

If your design is right, the answers to these should be quick:

- "What happens if the seed disk's signing key leaks?" → revoke the public
  key on the broker, re-mint host signing keys. Existing seeds become
  invalid.
- "What happens if a tenant VM is moved between hosts?" → host re-mints the
  seed at next boot, broker reissues the same age key (because tenant scope,
  not host, determines the answer).
- "What happens if Janus is down?" → existing running VMs keep working
  (they have decrypted secrets in `/run/secrets`); newly-booted VMs fail to
  start until Janus is back. This is the same availability characteristic
  as needing a working KMS.
- "What happens if a VM gets cloned?" → the clone has the same seed and the
  same token, so the broker rejects the second use (single-use). The clone
  needs a fresh seed.
- "What happens if an attacker compromises a tenant VM?" → they get the
  current pool's secrets. They cannot escalate to a different pool's
  secrets, because the broker only issues the one this VM identifies as.
  Revocation = remove that pool's recipient, rotate, redeploy.

If any of those answers are bad, the design has a hole. As sketched, they're
all defensible.
